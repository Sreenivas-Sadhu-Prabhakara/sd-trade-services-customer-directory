Each service domain is its own standalone Spring Boot repository — clone and run in seconds, or deploy onto Kubernetes with the bundled Helm chart behind the Cilium mesh.

# run this service domain standalone
git clone https://github.com/Sreenivas-Sadhu-Prabhakara/sd-trade-services-customer-directory
mvn -f sd-trade-services-customer-directory spring-boot:run
curl localhost:8080/v1/service-domain

# or deploy onto Kubernetes (Cilium eBPF mesh)
helm upgrade --install sd-trade-services-customer-directory ./helm -n bian-operations

FAQQuestions

What is the BIAN “Catalog” functional pattern?
A Catalog service domain maintains an authoritative directory or reference list, serving golden-source records to the rest of the bank. Trade Services Customer Directory applies it to the Trade Counterparty.
What is a BIAN service domain?
BIAN (the Banking Industry Architecture Network) decomposes a bank into independent service domains, each owning exactly one business capability. Trade Services Customer Directory is the service domain for Trade Banking; it manages the Trade Counterparty Directory Entry and exposes it through standard BIAN action terms.
What business area does Trade Services Customer Directory belong to?
Trade Services Customer Directory sits in the Operations and Execution business area — the operational core that executes payments, positions, settlements and transactions. It runs in the bian-operations namespace as one independent Spring Boot microservice.
Is Trade Services Customer Directory production-ready?
It is an educational reference implementation: a real BIAN REST API over an in-memory store, ready to be filled with domain logic. It is not a live bank and is not affiliated with BIAN.
